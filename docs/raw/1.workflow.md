Ок — тогда режем варианты до “рельс” и строим **2–3 суперпростых workflow**, где агент делает почти всё, а команда делает минимум, но при этом у тебя всегда есть понятная картинка “что происходит”.

Ниже — компактная система, ориентированная именно на автономных агентов.

---

## 0) Главная идея: 3 состояния + 2 очереди + 1 точка контроля

### Состояния (labels на issue)

* `status:backlog` — входящие (сырьё)
* `status:todo` — одобрено к выполнению
* `status:in-progress` — выполняется агентом/разработчиком
  Done = закрыто (и автоматически `status:done`)

### Две очереди

* **Backlog** = всё что пришло (идеи/баги/запросы)
* **Todo** = “разрешено делать” (контроль команды)

### Одна точка контроля команды

Команда **не управляет процессом вручную**, она только:

* переводит задачи **Backlog → Todo** (approve)
* и делает review PR (approve merge)

Это даёт максимум контроля при минимуме действий.

---

## 1) Режем “типы” полностью: один унифицированный формат issue

Мы оставляем только:

* `status:*`
* заголовок `kind(area): ...` (как вы уже делаете)
* в теле — `Spec: R-XXX` или `Spec: New requirement`

Для упрощения для агентов — **в Todo всегда должен быть Spec**:

* даже для багов
* даже для мелких фич
* исключение: чистый `chore(...)`

Итого агенту не нужно гадать: *“где правда?”* — всегда в `Spec:`.

## 2) Три простых workflow (вместо десятков)

### Workflow A — “Запрос / идея / баг” (создание карточки)

**Команда делает:** 0 действий
**Агент делает:** 100%

**Вход:** “добавь фичу …” / “зафиксируй баг …”
**Выход:** issue в `status:backlog` (всегда)

**Правило:** всё новое сначала backlog.
Никаких todo автоматически — это и есть контроль.

---

### Workflow B — “Approve to build” (контроль и прозрачность)

**Команда делает:** 1 действие (approve)
**Агент делает:** дальше всё

**Смысл:** перевод в `status:todo` — это “разрешение агенту работать”.

**Definition of Ready для Todo (сильно упрощённый):**
В issue должно быть всего 3 вещи:

1. `Spec: R-XXX` или `Spec: New requirement`
2. `Goal:` 1–3 пункта
3. `Acceptance:` 2–6 чекбоксов

Если этого нет — агент сам дополняет issue (задаёт вопросы или предлагает варианты), но в `todo` переводит только команда.

---

### Workflow C — “Implement & PR” (агент делает всю работу)

**Команда делает:** review PR + merge
**Агент делает:** ветка → код → тесты → PR → (опционально) обновление spec

**Правило:** агент берёт только `status:todo`.

Пайплайн агента:

1. взять issue (перевести в `status:in-progress`, назначить себя)
2. создать ветку
3. прочитать `spec.md` по `R-XXX`
4. реализовать + тесты
5. открыть PR с:

   * `Fixes #123`
   * `Spec: R-XXX`
6. дождаться CI
7. подготовить self-review notes

## 3) Что даёт тебе “понятно, что происходит сейчас”

Ты хочешь одну картинку “в данный момент”. Делается так:

### 3.1 Один “операционный экран” — три saved search

* **Queue (вход):** backlog
  `is:issue is:open label:status:backlog`
* **Approved:** todo
  `is:issue is:open label:status:todo`
* **Running:** in-progress
  `is:issue is:open label:status:in-progress`

И отдельный:

* **Waiting review:** PRs open
  `is:pr is:open`

Это уже почти “панель управления” без Projects.

### 3.2 Жёсткое правило WIP

Чтобы было контролируемо:

* агент **не берёт больше N задач одновременно**
* N=1 или N=2 (для начала)

Это сильно повышает предсказуемость.

---

## 4) Ультра-упрощение Spec процесса для агентов

Чтобы агент не делал “архитектуру ради архитектуры”:

### Единственное место требований: `spec.md`

* новые требования добавляются в конец
* для изменения — правим R-XXX

### Два режима работы агента со spec

* **Spec-first** (если `Spec: New requirement`)
  агент создаёт PR, который **только добавляет R-XXX**, и ждёт merge
* **Implement** (если `Spec: R-XXX`)
  агент реализует и делает PR на код

Это разделяет риски: команда видит изменения требований отдельно от кода.

---

## 5) Минимум действий команды при максимальном контроле

### Команда делает только:

1. периодически просматривает backlog и переводит нужное в todo
2. смотрит PR и мержит

### Всё остальное делает агент:

* уточняет требования
* пишет spec PR
* пишет код PR
* ведёт статус
* пишет комментарии “progress update”

---

## 6) Автоматизация контроля (чтобы процесс сам держался)

У вас уже есть enforcement:

* PR с изменениями `src/`/`tests/` требует `Fixes #` и `Spec:`

Добавь ещё 2 “автоматических страховки” (очень маленькие):

### 6.1 Автоматический комментарий “progress update”

Когда issue переводится в `status:in-progress`, агент обязан оставить 1 комментарий:

* что делает
* какие файлы трогает
* ожидаемое время до PR (без оценок времени можно просто “в процессе/готовлю PR”)

Это даёт тебе “живую ленту”.

### 6.2 Авто-правило: нельзя переводить в todo без Spec/Acceptance

Либо CI, либо action на issues (если хочешь) — но можно и просто договором.

---

## 7) Самое простое “API” для агентов: 4 команды верхнего уровня

Вместо 12 skills — четыре “макро-команды”:

1. **`capture`** — создать backlog issue (idea/bug/feature request)
2. **`prepare`** — довести backlog issue до “готов к approve” (добавить Spec/Acceptance, задать вопросы)
3. **`execute`** — взять todo и сделать PR (spec-first или implement)
4. **`report`** — дать состояние: backlog/todo/in-progress/pr

Под капотом они вызывают мелкие actions, но для агентов это суперпростая модель.

---

## 8) Предлагаю финальный “режим” для вашего проекта (на старте)

**Рекомендую выбрать строго такой режим:**

* Любая новая фича = сначала `spec-first` PR (добавить/изменить R-XXX)
* Потом `implement` PR

Да, это медленнее на 1 PR, но:

* тебе всегда ясно, *что именно меняется*
* агент меньше ошибается
* контроль команды максимальный
