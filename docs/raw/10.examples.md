Ниже — несколько “реальных” примеров задач в вашем формате **one-PR + Plan Gate + tasks/**. Я покажу для каждой:

* Issue (title + ключевые поля body)
* `tasks/T-XXXX.md` (план)
* изменения в `spec.md` (если нужны)
* ветка/коммиты
* PR body

Числа (`#123`, `T-0123`, `R-0XX`) — примерные.

## Пример 1 — Фича с существующим требованием

### Контекст

В `spec.md` уже есть:

`R-021: API MUST allow exporting run results as JSON.`

### Issue (создаётся сразу как рабочая)

**Title:** `feat(api): implement R-021 json export endpoint`
**Labels:** `status:backlog`

**Body (коротко):**

```md
Spec: R-021
Plan: tasks/T-0123.md

Why:
- Нужен экспорт результатов для интеграций

Goal:
- Добавить endpoint /runs/{id}/export.json

Acceptance:
- [ ] Endpoint возвращает JSON по схеме
- [ ] 404 если run отсутствует
- [ ] Интеграционный тест покрывает happy-path и 404
```

### Plan commit: `tasks/T-0123.md`

```md
# T-0123: JSON export endpoint

Issue: #123
Spec: R-021

Why:
- Автоматизация интеграций и отчётности

Goal:
- Экспортировать результаты run в JSON

Plan:
- [ ] Добавить handler GET /runs/{id}/export.json
- [ ] Определить/зафиксировать JSON schema (reuse existing DTO)
- [ ] Добавить 404/validation
- [ ] Добавить integration test (happy-path + 404)
- [ ] Обновить README (если endpoint публичный)

Acceptance:
- [ ] JSON валиден и стабилен
- [ ] Тесты проходят локально и в CI
```

### Ветка

`feat/123-json-export`

### Коммиты (в одном PR)

1. `plan(api): T-0123 define plan for R-021 json export`
2. *(после plan:approved)* `feat(api): R-021 implement json export endpoint`
3. `test(api): R-021 add integration coverage`

### PR body

```md
Fixes #123
Spec: R-021
Plan: tasks/T-0123.md

Why:
- Экспорт нужен для внешних интеграций и автоматизации отчётов
```

## Пример 2 — Баг с существующим требованием

### Контекст

`R-002: STM MUST be cleared between tasks.`

### Issue

**Title:** `fix(memory): STM not cleared after tool failure`
**Labels:** `status:backlog`

**Body:**

```md
Spec: R-002
Plan: tasks/T-0124.md

Why:
- Утечка контекста ломает предсказуемость агентов

Observed:
- После tool error следующий task видит прошлые сообщения

Expected:
- STM очищена при старте каждой задачи

Acceptance:
- [ ] Добавлен регрессионный тест на tool failure
- [ ] STM очищается даже при исключении/ошибке инструмента
```

### `tasks/T-0124.md`

```md
# T-0124: Clear STM after tool failure

Issue: #124
Spec: R-002

Plan:
- [ ] Воспроизвести: task -> tool error -> next task reads stale STM
- [ ] Добавить failing regression test
- [ ] Найти место очистки STM (task boundary)
- [ ] Гарантировать cleanup в finally / deferred hook
- [ ] Прогнать тесты и убедиться, что нет регрессий

Acceptance:
- [ ] Регрессионный тест падает до фикса и проходит после
- [ ] Очистка происходит на всех exit paths
```

### Ветка

`fix/124-stm-tool-failure`

### Коммиты

1. `plan(memory): T-0124 plan fix for R-002 tool failure leak`
2. *(после plan:approved)* `test(memory): R-002 add regression for tool failure boundary`
3. `fix(memory): R-002 ensure STM cleanup in finally`

### PR body

```md
Fixes #124
Spec: R-002
Plan: tasks/T-0124.md

Why:
- Утечка контекста делает поведение агента непредсказуемым и опасным
```

## Пример 3 — “Баг”, но в spec нет R-0YY (узакониваем ожидание)

### Ситуация

Пользователь жалуется: агент иногда повторно использует state прошлой задачи. В spec про это ничего нет.

### Issue

**Title:** `fix(agent): cross-task state carryover in planner`
**Labels:** `status:backlog`

**Body:**

```md
Spec: Unknown
Plan: tasks/T-0125.md

Why:
- Непредсказуемые ответы, сложно отлаживать, риск утечек

Observed:
- В задаче B появляются решения/переменные из A

Expected:
- Состояние задачи изолировано (если не сохранено явно)
```

### Plan + Spec commit (в том же PR, до кода)

`tasks/T-0125.md`:

```md
# T-0125: Task boundary state isolation

Issue: #125
Spec: New requirement (will become R-034)

Plan:
- [ ] Сформулировать требование: что такое "task boundary"
- [ ] Добавить R-034 в spec.md
- [ ] Добавить regression test на carryover
- [ ] Найти источник состояния (planner cache / global singleton)
- [ ] Очистка/изоляция состояния на boundary
- [ ] Подтвердить тестами

Acceptance:
- [ ] Есть требование R-034
- [ ] Есть regression test: fail->pass
- [ ] Нет carryover без явного persist
```

`spec.md` добавляется в конец:

```md
## R-034
Agent MUST NOT reuse prior task state across tasks unless explicitly persisted via LTM tool.
```

### Ветка

`fix/125-task-boundary-isolation`

### Коммиты

1. `plan(agent): T-0125 propose R-034 and plan`
2. `spec(agent): add R-034 task boundary isolation`
3. *(после plan:approved)* `test(agent): R-034 add regression for state carryover`
4. `fix(agent): R-034 clear planner state at task boundary`

### PR body

```md
Fixes #125
Spec: R-034
Plan: tasks/T-0125.md

Why:
- Без контракта “границы задачи” нельзя стабильно тестировать и предотвращать утечки
- Это снижает качество и предсказуемость агентов
```

## Пример 4 — Spec-only (без кода)

### Ситуация

Нужно уточнить требование к инструментам, чтобы агенты валидировали schema.

### Issue

**Title:** `spec(tools): require tool output schema validation`
**Labels:** `status:backlog`

**Body:**

```md
Spec: New requirement
Plan: tasks/T-0126.md

Why:
- Снижаем ошибки парсинга и галлюцинации форматов
```

### `tasks/T-0126.md`

```md
# T-0126: Tool output schema validation requirement

Issue: #126
Spec: New requirement (R-035)

Plan:
- [ ] Добавить R-035 в spec.md
- [ ] Уточнить термин "schema validation"
- [ ] Определить минимальные ожидания (fail fast vs fallback)
- [ ] Acceptance: однозначная формулировка

Acceptance:
- [ ] R-035 добавлено и понятно
```

`spec.md`:

```md
## R-035
Agent MUST validate tool output against the declared schema before using it.
```

### PR

Коммиты:

1. `plan(tools): T-0126 propose R-035`
2. `spec(tools): add R-035 tool schema validation`

После `plan:approved` можно сразу merge (без implement phase).

PR body:

```md
Fixes #126
Spec: R-035
Plan: tasks/T-0126.md

Why:
- Формализуем проверку инструментов для снижения ошибок и нестабильности
```

## Пример 5 — Chore (рефакторинг/инфра), без spec

### Issue

**Title:** `chore(infra): speed up CI by caching dependencies`
**Labels:** `status:backlog`

**Body:**

```md
Plan: tasks/T-0127.md

Why:
- Ускорить фидбек-луп и снизить стоимость CI

Acceptance:
- [ ] CI время снизилось минимум на 30%
- [ ] Нет нестабильности/флейков
```

### `tasks/T-0127.md`

```md
# T-0127: CI dependency caching

Issue: #127

Plan:
- [ ] Добавить cache step (key includes lockfile hash)
- [ ] Проверить hit-rate на 3 прогонах
- [ ] Зафиксировать до/после в Notes

Acceptance:
- [ ] Среднее время CI снизилось
```

### Коммиты

1. `plan(infra): T-0127 CI caching plan`
2. *(после plan:approved)* `chore(infra): add dependency cache to CI`
3. `docs(infra): note CI caching behavior` (если нужно)

PR body:

```md
Fixes #127
Plan: tasks/T-0127.md

Why:
- Снижаем время CI и ускоряем разработку
```
