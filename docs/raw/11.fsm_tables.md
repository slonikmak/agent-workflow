## 1) Состояния (state summary)

| State | Название               |                                                 Issue status | PR                                                            | Что разрешено                                |
| ----- | ---------------------- | -----------------------------------------------------------: | ------------------------------------------------------------- | -------------------------------------------- |
| S0    | Intake                 |                                                            — | нет                                                           | только входящий запрос                       |
| S1    | Backlog                |                                             `status:backlog` | нет или plan-only PR                                          | оформление задачи, уточнение                 |
| S2    | Plan PR Open           |                                             `status:backlog` | открыт, **без** `plan:approved`                               | только `tasks/`, `spec.md`, `adr/`           |
| S2a   | Plan Changes Requested |                                             `status:backlog` | открыт + `plan:changes-requested`                             | только правки плана/spec/ADR                 |
| S3    | Plan Approved          |                                                `status:todo` | открыт + `plan:approved`                                      | можно начинать реализацию                    |
| S4    | Implementing           |                                         `status:in-progress` | открыт + `plan:approved`                                      | код, тесты, docs/spec (если нужно)           |
| S5    | Review                 |                                         `status:in-progress` | открыт + `plan:approved`, CI зелёный                          | review fixes, финальное согласование         |
| S6    | Done                   |                                     `closed` + `status:done` | merged                                                        | завершено                                    |
| SX    | Replan Required        | `status:backlog` (или `status:in-progress`, если оставляете) | открыт, `plan:approved` снят, обычно `plan:changes-requested` | обновление плана/spec, повторное утверждение |

## 2) Таблица переходов (основной автомат)

| From | Event                  | Кто инициирует        | Условия                              | Действия                                                                                                                           | To  |
| ---- | ---------------------- | --------------------- | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------- | --- |
| S0   | `capture_request`      | человек/агент         | есть запрос                          | создать issue (`idea/feat/fix/spec/chore`) с `status:backlog`                                                                      | S1  |
| S1   | `propose_plan`         | агент/человек         | issue существует                     | нормализовать заголовок; создать ветку; добавить `tasks/T-XXXX.md`; при необходимости обновить `spec.md`/`adr/`; открыть PR        | S2  |
| S2   | `request_plan_changes` | команда               | план/спека требуют правок            | поставить `plan:changes-requested` (и убедиться, что `plan:approved` снят)                                                         | S2a |
| S2a  | `update_plan`          | агент/человек         | получены замечания                   | обновить `tasks/` и/или `spec.md`/`adr/`; ответить в PR                                                                            | S2  |
| S2   | `approve_plan`         | команда               | PR содержит корректный plan/spec     | снять `plan:changes-requested` (если есть), поставить `plan:approved`; issue → `status:todo`                                       | S3  |
| S2a  | `approve_plan`         | команда               | правки внесены                       | снять `plan:changes-requested`, поставить `plan:approved`; issue → `status:todo`                                                   | S3  |
| S3   | `start_implementation` | агент/человек         | `plan:approved` есть                 | issue → `status:in-progress`; коммиты в `src/`/`tests/`                                                                            | S4  |
| S4   | `ready_for_review`     | агент/человек         | CI зелёный, Acceptance выполнен      | обновить PR описание/self-review                                                                                                   | S5  |
| S5   | `merge`                | команда               | review ok                            | merge PR; issue закрывается (`Fixes #`)                                                                                            | S6  |
| S4   | `scope_change`         | агент/человек         | план/спека существенно меняются      | обновить `tasks/` и/или `spec.md`; снять `plan:approved`; поставить `plan:changes-requested`; issue → `status:backlog` (рекоменд.) | SX  |
| S5   | `scope_change`         | агент/человек/ревьюер | в review выявлено изменение scope    | обновить `tasks/` и/или `spec.md`; снять `plan:approved`; поставить `plan:changes-requested`; issue → `status:backlog` (рекоменд.) | SX  |
| SX   | `update_plan`          | агент/человек         | замечания понятны                    | правка `tasks/`/`spec.md`/`adr/`                                                                                                   | S2a |
| SX   | `approve_plan`         | команда               | обновлённый план ок                  | снять `plan:changes-requested`, поставить `plan:approved`; issue → `status:todo`                                                   | S3  |
| S3   | `request_plan_changes` | команда               | перед стартом нашли проблему в плане | снять `plan:approved`, поставить `plan:changes-requested`; issue → `status:backlog`                                                | S2a |

## 3) Что должно быть в каждом состоянии (инварианты)

| State                      | Обязательные артефакты                                     | Labels на PR                                                          | Запреты                                                             |
| -------------------------- | ---------------------------------------------------------- | --------------------------------------------------------------------- | ------------------------------------------------------------------- |
| S1 Backlog                 | issue                                                      | —                                                                     | нет обещания “готово делать”                                        |
| S2 Plan PR Open            | PR + `tasks/T-XXXX.md`; `spec.md`/`adr/` при необходимости | **нет** `plan:approved`; `plan:changes-requested` может отсутствовать | **нельзя** менять `src/`/`tests/`                                   |
| S2a Plan Changes Requested | всё как в S2                                               | `plan:changes-requested`                                              | **нельзя** менять `src/`/`tests/`; нельзя ставить `status:todo`     |
| S3 Plan Approved           | `tasks/T-XXXX.md`; issue body с `Plan:` и `Spec:`          | `plan:approved` (и нет `plan:changes-requested`)                      | нельзя терять связь с issue/spec/plan                               |
| S4 Implementing            | PR body: `Fixes`, `Spec`, `Plan`, `Why`                    | `plan:approved`                                                       | нельзя менять scope без снятия approval                             |
| S5 Review                  | CI зелёный; Acceptance выполнен                            | `plan:approved`                                                       | нельзя merge, если появились новые scope/spec изменения без re-plan |
| S6 Done                    | issue closed + `status:done`                               | merged PR                                                             | —                                                                   |
| SX Replan Required         | обновлённый `tasks/` и/или `spec.md` в процессе            | обычно `plan:changes-requested`, `plan:approved` снят                 | нельзя продолжать кодовые изменения до повторного approve           |

## 4) Как FSM покрывает типы задач (с учётом `plan:changes-requested`)

| Тип задачи                  | Что происходит в Plan phase (S2 / S2a)                                                                             | Что происходит в Implement phase (S4) |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------ | ------------------------------------- |
| Фича с существующим `R-XXX` | создаётся `tasks/T-XXXX.md`; при замечаниях команды → `plan:changes-requested` и правки плана                      | код + тесты                           |
| Фича без `R-XXX`            | `tasks/T-XXXX.md` + новый `R-0YY` в `spec.md`; часто возможны правки формулировки через `plan:changes-requested`   | код + тесты                           |
| Баг с `R-XXX`               | `tasks/T-XXXX.md` (обычно с шагом “регрессионный тест”); если план фикса не ок — `plan:changes-requested`          | тест (fail) → фикс → тест (pass)      |
| Баг без `R-XXX`             | `tasks/T-XXXX.md` + новый `R-0YY` в `spec.md` (узакониваем ожидание); команда может вернуть формулировку на правки | тест + фикс                           |
| Spec-only                   | `tasks/T-XXXX.md` + `spec.md`; цикл правок через `plan:changes-requested` очень типичен                            | S4 можно пропустить                   |
| Chore                       | `tasks/T-XXXX.md`; при спорном плане/оценке эффекта — `plan:changes-requested`                                     | код/CI/docs без `Spec`                |

## 5) Нормализация заголовка issue (правило)

| Когда                              | Разрешённый заголовок                            | Комментарий                    |
| ---------------------------------- | ------------------------------------------------ | ------------------------------ |
| В S1 (сырой вход)                  | можно `idea(area): ...`                          | это ещё не контракт            |
| При переходе в S2 (`propose_plan`) | **обязательно** `feat/fix/spec/chore(area): ...` | заголовок становится “рабочим” |
| В S2a (`plan:changes-requested`)   | тот же рабочий заголовок                         | не возвращаемся к `idea(...)`  |
| В S3+ (`todo` и дальше)            | `feat/fix/spec/chore(area): ...`                 | `idea(...)` не допускается     |

## 6) Минимальный “операционный экран” по FSM

| Что смотреть                                                       | Значение                                            |
| ------------------------------------------------------------------ | --------------------------------------------------- |
| `status:backlog` + **без PR**                                      | сырые входящие (S1)                                 |
| open PR **без** `plan:approved` и **без** `plan:changes-requested` | план предложен, ждёт первого review (S2)            |
| open PR с `plan:changes-requested`                                 | план требует доработки (S2a / SX)                   |
| `status:todo` + PR с `plan:approved`                               | план утверждён, задача готова/может стартовать (S3) |
| `status:in-progress` + PR с `plan:approved`                        | реализация идёт / ревью (S4/S5)                     |
| closed issues                                                      | завершённые задачи (S6)                             |

## 7) Рекомендуемые labels для PR (итог)

| Label                    | Назначение                       | Взаимоисключаемость                                 |
| ------------------------ | -------------------------------- | --------------------------------------------------- |
| `plan:approved`          | план/спека утверждены, код можно | не должен сосуществовать с `plan:changes-requested` |
| `plan:changes-requested` | план/спека требуют правок        | не должен сосуществовать с `plan:approved`          |

### Простое правило

На PR в Plan phase должен быть **не более одного** из:

* `plan:approved`
* `plan:changes-requested`

(или ни одного — если ещё никто не посмотрел)
