Я отвечу как **world-famous PhD по software architecture и AI-assisted engineering**, лауреат **ACM SIGSOFT Distinguished Paper Award**.

Это отличный кейс — и он **нормально ложится** в ваш процесс. Просто тут важно разделить два типа работ:

1. **Изменение поведения системы** → требует `spec.md` (`R-XXX`)
2. **Изменение архитектуры/техдолга без изменения поведения** → это **не spec**, а **plan + ADR/chore**

То есть для архитектуры и техдолга у вас появляется отдельный “трек” внутри того же FSM, без ломания модели.

---

## Ключевая идея

### Если вы делаете “аудит → потом имплемент”

Это не одна фича, а **две фазы** (и обычно лучше две задачи):

### Вариант (рекомендую)

* **Issue A:** `chore(...)` или `spec(...)` на **аудит/дизайн**
* **Issue B:** `chore(...)` / `feat(...)` на **реализацию** (создаётся по итогам аудита)

Так у вас:

* контроль лучше
* план реалистичнее
* агент не “прыгает в код” до понимания картины

# Как это ложится на процесс (общая схема)

## Сценарий: “Доработать архитектуру” (audit-first)

### Этап 1 — Audit / Design (отдельная задача)

**Тип issue:** обычно `chore(area): architecture audit ...`
(или `spec(...)`, если точно знаете, что изменится поведение/контракт)

**Что в PR:**

* `tasks/T-XXXX.md` (план аудита)
* `adr/000X-...md` (если уже есть решение)
* возможно `docs/architecture.md` (если ведёте)
* **без `src/`/`tests/`**

**Результат merge:**

* есть выводы аудита
* есть ADR (если нужно)
* есть список конкретных реализационных задач (новые issues)

➡️ После этого агент/команда создаёт 1+ новых issue на имплементацию.

---

### Этап 2 — Implement (обычные задачи)

Дальше всё как у обычных кейсов:

* если поведение меняется → `Spec: New requirement` / `R-XXX`
* если поведение не меняется → `chore(...)`
* один PR на задачу с plan gate

---

# Почему лучше не “аудит + имплемент” в одном issue/PR

Можно, но обычно хуже:

* план аудита почти всегда меняет scope
* сложно заранее утвердить реализационный plan
* PR становится длинным и менее контролируемым

Для автономных агентов особенно важно:

* **сначала диагностический PR**
* потом **реализационные PR**

Это делает агентов предсказуемыми.

# Конкретный кейс 1: Доработка архитектуры (audit → implement)

## Шаг A1 — Audit issue

**Title:** `chore(agent): audit planner/executor architecture boundaries`
**Labels:** `status:backlog`

**Issue body:**

```md
Plan: tasks/T-0201.md

Why:
- Есть признаки смешения ответственности planner/executor
- Это мешает масштабировать новые tools и отлаживать поведение

Goal:
- Провести аудит текущей архитектуры
- Подготовить предложение по изменениям
- Сформировать список реализационных задач

Acceptance:
- [ ] Есть audit report (в tasks/T-0201.md или docs/)
- [ ] Есть ADR (если требуется архитектурное решение)
- [ ] Созданы отдельные issue на реализацию
```

## Шаг A2 — Plan PR (audit-only)

**Ветка:** `chore/201-architecture-audit`

**Коммит #1 (plan):**

* `tasks/T-0201.md`

**Коммит #2 (audit artifacts):**

* `docs/architecture-audit-2026-02.md` *(опционально)*
* `adr/0007-planner-executor-boundary.md` *(если решение найдено)*

**PR body:**

```md
Fixes #201
Plan: tasks/T-0201.md

Why:
- Нужно сначала зафиксировать проблемы и решение, прежде чем менять код
```

### Plan Gate

Команда смотрит план аудита → `plan:approved` → issue `status:todo`

### Implement phase?

Для audit-only задачи **кодовая фаза может не быть вообще**.
После подготовки audit/ADR PR идёт в review и merge.

## Шаг A3 — После merge: создание реализационных задач

По результатам аудита агент создаёт, например:

* `chore(agent): split planner/executor modules`
* `feat(agent): implement execution context object (R-041)` *(если меняется контракт)*
* `chore(test): add architecture regression tests`

Каждая — уже отдельный one-PR workflow.

# Конкретный кейс 2: Техдолг (без изменений требований)

Это обычно **чистый `chore(...)`**. `spec.md` не нужен.

## Пример: “разделить большой сервис памяти”

### Issue

**Title:** `chore(memory): split memory service into stm/ltm modules`
**Labels:** `status:backlog`

**Body:**

```md
Plan: tasks/T-0202.md

Why:
- Файл слишком большой, сложно вносить изменения
- Высокий риск побочных эффектов

Goal:
- Разделить код на модули STM/LTM без изменения поведения

Acceptance:
- [ ] Поведение не изменилось (все текущие тесты зелёные)
- [ ] Структура модулей отражает домены STM/LTM
- [ ] Нет публичных API-изменений
```

### `tasks/T-0202.md`

```md
# T-0202: Split memory service modules

Issue: #202

Plan:
- [ ] Зафиксировать текущие public interfaces (чек-лист)
- [ ] Вынести STM logic в отдельный модуль
- [ ] Вынести LTM logic в отдельный модуль
- [ ] Обновить импорты и wiring
- [ ] Прогнать unit/integration tests
- [ ] Проверить отсутствие изменений публичного поведения

Acceptance:
- [ ] Все тесты проходят
- [ ] Нет изменений API/контрактов
```

### PR

Коммиты:

1. `plan(memory): T-0202 refactor plan for service split`
2. *(после plan:approved)* `chore(memory): split memory service into stm and ltm modules`
3. `test(memory): verify no behavior regression after split` *(если добавили coverage)*

**PR body:**

```md
Fixes #202
Plan: tasks/T-0202.md

Why:
- Уменьшаем связанность и упрощаем дальнейшие изменения
```

# Конкретный кейс 3: Техдолг, который вскрывает изменение поведения

Это важный гибридный случай.

Например вы делаете `chore`, но в процессе понимаете, что без изменения контракта не обойтись.

### Что делать?

Срабатывает ваш FSM переход `scope_change`:

1. обновляете `tasks/T-XXXX.md`
2. добавляете/меняете `spec.md` (`R-XXX` или `New requirement`)
3. снимаете `plan:approved`
4. ставите `plan:changes-requested`
5. команда re-approves
6. продолжаете

То есть `chore` может превратиться в `feat/spec`:

* **и заголовок issue надо нормализовать**

  * было: `chore(agent): simplify context pipeline`
  * стало: `feat(agent): implement R-042 context pipeline contract`

Это нормально и даже полезно — значит процесс поймал изменение scope.

---

# Нужно ли вводить отдельный тип `audit`?

Можно, но я бы **не вводил** (чтобы не плодить сущности).

Используйте:

* `chore(...)` — для аудита/арх.работы без изменения поведения
* `spec(...)` — если задача именно про формализацию требований
* `feat/fix(...)` — когда уже код и поведение

Если очень хочется визуально отделять аудиты, можно добавить **label**:

* `work:audit`

Это лучше, чем новый `kind`.

# Как это выглядит в FSM (коротко)

## Аудит / техдолг без поведения

* S1 backlog
* S2 plan PR open (`tasks/` + docs/ADR)
* S2a если `plan:changes-requested`
* S3 plan approved (`todo`)
* S5 review (S4 можно почти пропустить, если нет кода)
* S6 done

## Аудит → реализация

* Первая задача завершается в S6
* По её результатам создаются новые issues
* Каждая новая задача проходит обычный цикл

---

# Практическое правило, которое я бы закрепил

### “Аудит не должен содержать кодовых изменений”

Если задача заявлена как аудит:

* PR не меняет `src/`/`tests/`
* только `tasks/`, `docs/`, `adr/`, `spec.md` (если это про требования)

Это держит процесс прозрачным:

* один PR = один смысл
* команде легко ревьюить
* агентам проще не “схлопывать” анализ и реализацию

---

# Итог

Да, кейсы “архитектура” и “техдолг” прекрасно ложатся на ваш процесс:

* **Техдолг без изменения поведения** → `chore(...)`, plan в `tasks/`, один PR
* **Архитектурная доработка с аудитом** → сначала отдельный `chore(...): audit ...` PR, потом реализационные задачи
* **Если в процессе всплыло изменение поведения** → `scope_change`, re-plan, `spec.md`, re-approve

Это как раз тот тип контролируемого, агент-friendly процесса, который хорошо живёт в маленькой команде.
