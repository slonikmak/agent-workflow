## 1) “Зачем рядом с изменением” — где именно это хранить

Самый практичный вариант на GitHub (и для агентов, и для людей):

### Уровень A (обязательный): PR body + связанный issue

Каждый PR, который меняет `src/`/`tests/`, обязан содержать:

* `Fixes #123`
* `Spec: R-XXX`
* **Why:** 2–5 строк “зачем” (мотивация/цель)

То есть в PR рядом с diff всегда есть текст “почему”.

Плюс: issue хранит Goal/Acceptance (контекст), а PR — “зачем именно этот diff”.

**Это уже почти идеально**: в истории GitHub у любого изменения есть link на причину.

### Уровень B (рекомендуемый): WHY в коммитах (очень коротко)

Чтобы “зачем” было ещё ближе к конкретным кускам кода, добавьте правило:

* Каждый коммит, который меняет поведение, начинается с `R-XXX:` и содержит 1 строку Why.

Пример коммита:

* `feat(memory): R-013 store task summary in LTM`
  и в теле:
* `Why: enable long-term recall across tasks`

Это помогает при `git blame` и просмотре истории без GitHub UI.

### Уровень C (только когда нужно): ADR

Если “зачем” — это выбор на развилке, а не просто мотивация, тогда ADR.
Но ADR — редкий случай, не для каждого изменения.

✅ Вывод: “зачем рядом” обеспечивается **PR body + коммит message**, а глубинное “почему так” — ADR.

## 2) Todo = утверждённый план — как это формализовать

Сейчас у вас `status:todo` — это “разрешено делать”.
Ты хочешь, чтобы это было “разрешено делать **по утверждённому плану**”.

Это решается одним правилом: **в Todo обязателен Plan**.

### Новый минимальный контракт для Todo-issue

Issue в `status:todo` должно содержать:

* `Spec: R-XXX` или `Spec: New requirement`
* `Goal` (что получаем)
* `Plan` (как делаем)
* `Acceptance` (как проверяем)

Вот шаблон:

```md
Kind: feat
Area: memory
Spec: R-013

Why:
- Зачем это нужно (1–3 пункта)

Goal:
- Что будет сделано (1–3 пункта)

Plan:
- [ ] Шаг 1: ...
- [ ] Шаг 2: ...
- [ ] Шаг 3: ...

Acceptance:
- [ ] ...
- [ ] ...
```

**Ключ:** Plan — это не “много текста”, а 3–7 чекбоксов.
Именно чекбоксы делают план машинно- и человеко-читабельным.

### Кто утверждает план?

Тут 2 режима (можно выбрать один):

**Режим строгого контроля (советую для старта):**

* Агент/разработчик готовит issue в backlog (или “готово к approve”)
* Команда переводит в `status:todo` **только после просмотра Plan**
* Это и есть “утверждение”

**Режим мягкий (для скорости):**

* Агент сам переводит в todo, но только если Plan заполнен
* Команда контролирует через PR review (меньше контроля, быстрее)

Ты просил максимальный контроль — значит лучше строгий режим.

## 3) Как это вписывается в текущие enforcement и automation

### PR enforcement уже есть

Вы требуете:

* `Fixes #123`
* `Spec: R-XXX`

Добавляем ещё одно требование:

* `Why:` в PR body

Это не ломает процесс и резко повышает трассировку.

### Issue flow остаётся прежним

* backlog → todo → in-progress → closed (+ status:done)

Только меняется Definition of Ready для `todo`:

* теперь включён `Plan`

---

## 4) Как выглядит стандартный путь теперь (очень кратко)

### Подготовка (кто угодно — агент или человек)

1. Создать issue (backlog)
2. Заполнить Spec + Why + Goal + Plan + Acceptance
3. Команда approves → переводит в `status:todo`

### Исполнение (агент или человек)

4. Берёт todo → in-progress
5. Ветка → код/тесты → коммиты с `R-XXX`
6. PR:

   * Fixes
   * Spec
   * Why
7. Merge → done

---

## 5) Почему это даёт тебе “понятно что происходит”

* `status:todo` — список задач, где уже есть **утверждённый план**
* Любой PR/коммит объясняет **зачем** и ссылается на требование
* Любой кусок кода можно связать:
  `git blame → коммит → PR → issue → R-XXX`

Это “идеальный” уровень контролируемости без досок и тяжёлых систем.
