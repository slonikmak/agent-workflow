Отлично — вынести план в репозиторий (в `tasks/`) очень хорошо усиливает твою цель “всё рядом с изменением” и делает процесс ещё более агент-дружелюбным и контролируемым.

Ниже — как это аккуратно уложить в воркфлоу, и ответ про заголовок `idea`.

---

## 1) План в репозитории: `tasks/` как “прикреплённый” артефакт

### Минимальная структура

```
tasks/
  T-0123.md
```

Где `123` — номер issue.

### Содержимое `tasks/T-0123.md`

Очень короткий и формальный формат (читается агентами и людьми):

```md
# T-0123: <short title>

Issue: #123
Spec: R-013

Why:
- ...

Goal:
- ...

Plan:
- [ ] Step 1: ...
- [ ] Step 2: ...
- [ ] Step 3: ...

Acceptance:
- [ ] ...
- [ ] ...

Notes:
- ...
```

**Правило:** если issue в `status:todo`, то соответствующий `tasks/T-XXXX.md` обязателен.

### Где “истина” про план?

* В issue остаётся короткая ссылка (1–2 строки)
* Подробности — в `tasks/T-XXXX.md`

В issue достаточно держать:

```md
Spec: R-013
Plan: tasks/T-0123.md
```

(и можно краткое Why/Goal продублировать в 1–2 строки, если хочется)

## 2) Как это меняет user stories (коротко)

### Backlog → Todo (утверждение плана)

Теперь это означает:

* создан/обновлён `tasks/T-XXXX.md`
* в issue добавлена ссылка на этот файл
* только после этого issue получает `status:todo`

### PR (кодовые изменения)

Если PR реализует задачу, он должен:

* ссылаться `Fixes #123`
* содержать `Spec: R-XXX`
* и часто удобно добавить строку:

  * `Plan: tasks/T-0123.md`
* `Why:` остаётся в PR body

То есть “зачем” и “как делали” живут рядом:

* Why — в PR
* Plan — в репо в `tasks/`
* Requirement — в `spec.md`

---

## 3) Нужно ли коммитить plan-файл отдельным PR?

Для максимального контроля есть два режима.

### Режим A (самый контролируемый, рекомендую сначала)

**Plan фиксируется отдельным PR до реализации.**

Flow:

1. backlog issue
2. агент готовит `tasks/T-XXXX.md` + (если нужно) обновляет `spec.md`
3. PR “Plan/Spec” → команда мерджит
4. только потом issue переводится в `status:todo`
5. агент реализует код отдельным PR

Плюсы: ты всегда видишь утверждённый план **до** кода.
Минусы: +1 PR.

### Режим B (быстрее)

Plan-файл добавляется **в том же PR**, что и код.
Тогда “Todo” означает “план утверждён”, но он попадает в репо чуть позже. Я бы так делал только если команда маленькая и доверяет агенту.

---

## 4) Enforcement: простое правило для CI (если захочешь)

Можно расширить CI так, чтобы:

* если issue в todo → должен существовать `tasks/T-XXXX.md`
* если PR меняет `src/`/`tests/` → в PR body должна быть ссылка на plan (или хотя бы наличие файла в diff)

Но даже без CI это легко держать дисциплиной: “todo без tasks-файла запрещён”.

## 5) Про заголовок `idea` при переходе в todo

Я бы закрепил простое правило:

### ✅ Да, заголовок меняется при переходе в todo

Потому что `idea` — это “мысль”, а `todo` — “контракт на выполнение”. Это разные сущности.

**Как именно менять:**

* Было: `idea(api): export results as json`
* Стало (после утверждения): `feat(api): implement R-0XX json export`

То есть:

* меняется `kind` (`idea` → `feat` / `fix` / `spec` / `chore`)
* в заголовке появляется ссылка на `R-XXX` (если применимо)

### Почему это важно

* по одному списку `status:todo` ты сразу видишь не “идеи”, а “работу”
* агенту проще: `kind` сигнализирует, что делать
* история не теряется: issue тот же, просто уточнился

Если тебе важно не “переписывать историю”, альтернативный вариант:

* не переименовывать issue
* а в todo создавать новое issue (feat/fix/spec), а idea закрывать ссылкой “Superseded by #…”
  Но это добавляет сущностей. Для минимализма лучше просто менять заголовок.

---

## 6) Минимальные правила в одном месте (итог)

1. `status:todo` ⇒ обязательно существует `tasks/T-XXXX.md`
2. Issue в todo содержит:

   * `Spec: R-XXX`
   * `Plan: tasks/T-XXXX.md`
3. PR содержит:

   * `Fixes #XXXX`
   * `Spec: R-XXX`
   * `Why: ...`
   * (желательно) `Plan: tasks/T-XXXX.md`
4. При переводе `idea` → `todo` переименовать заголовок в `feat/fix/spec(...)`
