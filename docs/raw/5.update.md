Я отвечу как **world-famous PhD по software architecture и AI-assisted engineering**, лауреат **ACM SIGSOFT Distinguished Paper Award**.

Оба варианта рабочие. Разница — в том, **где у вас точка контроля** и насколько вы хотите “чистую” историю. Для автономных агентов я бы выбирал по простому правилу:

* если вам важнее **контроль и предсказуемость** → **2 PR**
* если важнее **скорость и минимум трения** → **1 PR с первым коммитом “plan/spec”**

Ниже — как это ложится на ваш воркфлоу и что будет удобнее именно для тебя.

---

## Вариант 1: Один PR, первый коммит = Plan/Spec (рекомендую как базовый)

### Как это выглядит

1. В backlog есть issue `idea(...)` или сырое `feat(...)`
2. Агент создаёт ветку `feat/123-...`
3. Делает **первый коммит**:

   * `tasks/T-0123.md` (план)
   * при необходимости обновляет `spec.md` (новый R-XXX или правка)
4. Открывает PR **сразу**

   * PR body: `Fixes #123`, `Spec: ...`, `Why: ...`, `Plan: tasks/T-0123.md`
5. Дальше агент продолжает пушить в **тот же PR** код и тесты

### Почему это удобно

* “Зачем” и “как” появляются рядом с изменениями **с самого начала**
* Команда сразу видит **план + направление** и может остановить/скорректировать
* Меньше PR → меньше шума, проще жить маленькой команде
* Для агентов это “один контейнер работы”, меньше шансов ошибиться с привязками

### Главное правило контроля (чтобы не поехало)

Сделайте “двухфазный PR” как норму:

* **Фаза A (Plan Gate):** пока не одобрен план, агент **не пушит код** (или пушит только черновые заготовки, но не просит мерджить)
* Команда пишет в PR комментарий типа: `Plan approved ✅`
* **Фаза B (Implement):** только после этого агент добавляет реализацию

Это даёт контроль почти как 2 PR, но остаётся один PR.

## Вариант 2: Два PR (максимум контроля, больше формальности)

### Как это выглядит

1. PR#1: только `tasks/T-0123.md` (+ `spec.md` если нужно)
2. Мержите PR#1 → issue становится `status:todo`
3. PR#2: код + тесты (и иногда дополнительные уточнения)

### Плюсы

* Очень чистая история
* Можно чётко разделить “контракт” и “реализацию”
* Легче делать ревью требований отдельно

### Минусы

* Больше трения (особенно с агентами)
* Больше PR, больше переключений контекста
* Иногда сложно не вносить мелкие уточнения в plan/spec уже во втором PR

---

## Что лучше именно под ваши цели

Ты хочешь одновременно:

* автономность агентов
* простые процессы
* минимум действий команды
* максимальный контроль “что происходит”

Под это лучше всего подходит:

### ✅ Один PR, но с жёстким Plan Gate

То есть:

* агент делает PR сразу
* первый коммит = план (и spec если нужно)
* команда approve плана одним комментарием/ревью
* после approve агент пишет код в тот же PR

Команда делает минимум:

* 1 раз глянула план в PR
* 1 раз сделала финальный ревью кода перед merge (можно тем же ревью)

---

## Как это состыковать с вашими статусами

Сейчас у вас идея: `todo` = утверждённый план.

При “одном PR” есть тонкость: план уже в PR, но ещё не утверждён.

Я бы сделал так:

* Пока Plan не approved: issue остаётся `status:backlog`
* После комментария `Plan approved ✅`: агент (или команда) ставит `status:todo`
  (или даже сразу `status:in-progress`, если агент уже начал)

Это сохраняет семантику todo как “утверждённый план”.

---

## Небольшая настройка, которая делает это железобетонным

Добавьте к вашему CI ещё одну лёгкую проверку для PR:

* если в PR есть `tasks/T-XXXX.md`, но **нет маркера approve**, CI может быть жёлтым/необязательным
* или проще: используйте label на PR `plan:approved` (ставит команда одним кликом)

Тогда агент может автоматически:

* ждать label `plan:approved`
* только потом пушить кодовые коммиты

---

## Итог

**Да, удобно**: один PR, где первый коммит — plan/spec, а дальше этот же PR используется для реализации.
Для маленького проекта с агентами это чаще всего **лучший компромисс**.
