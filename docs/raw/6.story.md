Я отвечу как **world-famous PhD по software architecture и AI-assisted engineering**, лауреат **ACM SIGSOFT Distinguished Paper Award**.

Ок — ниже полный набор кейсов **в режиме “один PR на задачу”** с двухфазной моделью **Plan Gate → Implement**, так чтобы:

* “зачем” было рядом с изменениями,
* `todo` означал “план утверждён”,
* агент мог работать автономно,
* у команды было минимум действий (в идеале: *approve plan* + *merge*).

Я буду описывать для каждого кейса:

* что происходит с **issue** (заголовок/описание/статус),
* с **документами** (`tasks/`, `spec.md`, `adr/`),
* с **веткой/коммитами**,
* с **PR**.

---

# Общие правила режима “Один PR”

## Статусы issue

* `status:backlog` — план не утверждён (даже если PR уже открыт)
* `status:todo` — **Plan Approved** (контрольная точка)
* `status:in-progress` — реализуем (после plan gate)
* closed → done (и авто `status:done`)

## Артефакты в репозитории

* `tasks/T-XXXX.md` — план задачи (обязателен до `todo`)
* `spec.md` — требования `R-XXX`
* `adr/000X-*.md` — только для развилок

## PR как “контейнер работы”

Один PR включает:

1. **Plan commit**: `tasks/` (+ `spec.md`/ADR при необходимости)
2. **Implement commits**: код/тесты (+ возможные мелкие уточнения docs/spec)

## Plan Gate (точка контроля)

Команда делает минимальное действие:

* ставит label на PR: `plan:approved`
  *(или оставляет комментарий `Plan approved ✅` — но label проще автоматизировать)*

Правило: **до `plan:approved` агент не коммитит изменения в `src/`/`tests/`.**

# Обязательные форматы

## Issue title (до и после)

* В backlog допускается `idea(area): ...`
* При появлении PR агент **нормализует** заголовок issue в рабочий kind:

  * `feat(area): ...` / `fix(area): ...` / `spec(area): ...` / `chore(area): ...`

Чтобы у тебя в списках было понятно, что это за работа.

## Issue body (когда есть PR)

Минимум:

```md
Spec: R-XXX | New requirement
Plan: tasks/T-XXXX.md

Why:
- ...

Goal:
- ...

Acceptance:
- [ ] ...
- [ ] ...
```

> Plan подробно лежит в `tasks/`, в issue можно держать короткую сводку.

## PR body (обязательно)

```md
Fixes #XXXX
Spec: R-XXX | New requirement
Plan: tasks/T-XXXX.md

Why:
- ...
```

# Кейсы

## Кейc 1 — Новая фича (требования уже есть: `R-XXX` существует)

### 0) Вход

Пользователь/команда: “сделай фичу X”.

### 1) Issue (создание/нормализация)

**Сначала** (может быть):

* `idea(api): export results as json` + `status:backlog`

**После того как агент начал оформление**:

* агент переименовывает в:

  * `feat(api): implement R-0XX json export`

**Issue status:** остаётся `status:backlog` (план ещё не утверждён).

**Issue body** агент заполняет:

* `Spec: R-0XX`
* `Plan: tasks/T-0XXX.md` (появится после plan commit)
* Why/Goal/Acceptance (кратко)

### 2) Ветка + PR (Plan commit)

**Ветка:**

* `feat/XXXX-json-export`

**Коммит #1 (plan):**

* добавляет `tasks/T-XXXX.md`
* не трогает `src/`/`tests/`

**Commit message:**

* `plan(api): T-XXXX define implementation plan for R-0XX`

**Открывается PR** сразу после plan commit.

**PR body:**

* Fixes/Spec/Plan/Why (см. шаблон выше)

### 3) Plan Gate

**Команда:** ставит `plan:approved` на PR.
**Issue:** переводится в `status:todo` (утверждён план).
Можно автоматически агентом после появления label.

### 4) Implement phase

**Issue:** `status:in-progress` (агент/разработчик ставит при старте реализации)

**Коммиты (код):**

* `feat(api): R-0XX implement json export`
* `test(api): R-0XX add integration tests`

Если нужно — мелкие уточнения `tasks/T-XXXX.md` можно делать отдельным коммитом:

* `plan(api): T-XXXX refine step 2 edge cases`

### 5) Review → Merge

Команда проверяет:

* соответствие `R-0XX` и Acceptance
* тесты
* отсутствие “неутверждённых” изменений требований

Merge → issue закрывается → `status:done`.

## Кейc 2 — Новая фича (требований НЕТ: нужен новый `R-XXX`)

Это важно для автономности: агент не “придумывает” поведение в коде.

### 1) Issue

* `idea(memory): persist task summary` (backlog)

Агент быстро нормализует в:

* `feat(memory): implement summary persistence (new requirement)`
  или
* `spec(memory): define summary persistence requirement`
  (оба допустимы, но я рекомендую оставить `feat(...)`, если цель — быстро дойти до реализации в одном PR; requirements всё равно будут в plan/spec коммите)

**Issue body:**

* `Spec: New requirement`
* Why/Goal/Acceptance (по возможности)

### 2) Plan commit включает spec

Ветка:

* `feat/XXXX-summary-persistence`

**Коммит #1 (plan + spec draft):**

* `tasks/T-XXXX.md`
* `spec.md` — добавлен новый блок в конец: `R-0YY`
  *(агент выбирает следующий номер)*

Commit message:

* `plan(memory): T-XXXX propose R-0YY and plan`
  или
* `spec(memory): add R-0YY summary persistence`
  (если хотите, чтобы spec изменения были отдельным коммитом внутри того же PR — это даже лучше)

**PR открывается**.
В PR body пока можно писать:

* `Spec: New requirement` (или уже `Spec: R-0YY`, если R присвоен)

### 3) Plan Gate = утверждение требований + плана

Команда читает:

* `tasks/T-XXXX.md`
* новый `R-0YY` в `spec.md`

**Действие:** ставит `plan:approved` (или просит правки).
После approve:

* issue становится `status:todo`

### 4) Implement phase

Коммиты:

* `feat(memory): R-0YY store summary in LTM`
* `test(memory): R-0YY add regression coverage`

Если в процессе выяснилось, что R-0YY надо уточнить — правьте `spec.md` *в этом же PR* отдельным коммитом:

* `spec(memory): refine R-0YY wording for edge case X`

### 5) Merge

Merge закрывает issue.

## Кейc 3 — Баг (нарушение существующего требования)

### 1) Issue

Сначала может быть:

* `fix(memory): STM leaks between tasks` + backlog

Агент находит в `spec.md`:

* `R-002 STM MUST be cleared between tasks`

**Issue body обновляется:**

* `Spec: R-002`

### 2) PR plan commit

Ветка:

* `fix/XXXX-stm-clear`

Коммит #1:

* `tasks/T-XXXX.md` (план расследования + фикс)
* (опционально) добавляет тест-кейс в plan как обязательный шаг

Commit message:

* `plan(memory): T-XXXX fix plan for R-002 regression`

PR body:

* Fixes/Spec/Plan/Why

### 3) Plan Gate

Команда approves план (часто быстро).

Issue → `status:todo`.

### 4) Implement

Коммиты:

* `test(memory): R-002 add failing regression test`
* `fix(memory): R-002 clear STM on task boundary`
* `test(memory): R-002 make regression pass`

(Сначала “красный тест” — очень хорошо для контроля.)

### 5) Merge

Закрытие issue.

---

## Кейc 4 — Уточнение/изменение требования без кода (spec-only)

Иногда нужно, и один PR это отлично поддерживает.

### 1) Issue

* `spec(tools): clarify tool schema validation` (backlog)

### 2) PR plan commit

Ветка:

* `spec/XXXX-tool-schema`

Коммит #1:

* `tasks/T-XXXX.md`
* изменения `spec.md` (правка R-XXX или новый R-YYY)

PR открыт.

### 3) Plan Gate

Тут Plan Gate = фактически review спеки.
Команда approves → `plan:approved`.
Issue можно прямо закрыть после merge, без in-progress.

### 4) Merge

Issue закрыт.

---

## Кейc 5 — Chore/Refactor (без изменения поведения)

Здесь spec не нужен, но “зачем” всё равно нужен.

### 1) Issue

* `chore(infra): speed up CI caching` (backlog)

### 2) PR plan commit

Коммит #1:

* `tasks/T-XXXX.md` (план: что делаем, как проверяем ускорение)
  PR открыт, Why обязателен.

### 3) Plan Gate

Команда approves (быстро).

### 4) Implement

Коммиты:

* `chore(infra): adjust cache keys`
* `test(infra): ensure CI still runs`

### 5) Merge

---

## Кейc 6 — “Изменение плана по ходу” (важно для автономности)

Если в реализации агент понял, что Plan надо менять существенно:

**Правило:** если изменение плана влияет на Acceptance/Scope — требуется повторный Plan Gate.

Как сделать без второго PR:

1. агент коммитит изменение `tasks/T-XXXX.md` + при необходимости `spec.md`
2. снимает label `plan:approved` (или ставит `plan:re-approve`)
3. команда снова approves
4. агент продолжает код

Это сохраняет контроль, не вводя второй PR.

# Что происходит с заголовком issue при переходах (строго)

### Backlog

* может быть `idea(area): ...`
* или сразу `feat/fix/...` (если пользователь точно сформулировал)

### Когда появляется PR (plan commit)

Я рекомендую **нормализовать заголовок всегда**, чтобы списки были понятны:

* `idea(...)` → `feat(...)` / `fix(...)` / `spec(...)` / `chore(...)`

### При переходе в todo

Заголовок уже должен быть “рабочим”. В todo **не должно** оставаться `idea(...)`, иначе трудно читать очередь.

# Минимальный набор файлов, который создаётся почти всегда

Для любых задач, кроме мелких trivial docs:

* `tasks/T-XXXX.md` — появляется первым коммитом в PR
* `spec.md` — появляется в PR, если:

  * `Spec: New requirement`, или
  * меняем существующий `R-XXX`
* `adr/000X-...md` — только если реальная развилка

---

# Как выглядит “полная картина” в любой момент

Ты смотришь:

1. `status:backlog` — входящие и PR без plan approval
2. PRs open + label `plan:approved` — готово к реализации/в реализации
3. `status:todo` — утверждённые планы (если PR уже открыт, это “approved and queued/running”)
4. `status:in-progress` — активно делается
5. merge history — всегда с Why/Spec/Plan

И любое изменение кода:

* имеет PR с Why
* привязано к issue (Fixes)
* привязано к требованию `R-XXX`
* имеет план в `tasks/`
