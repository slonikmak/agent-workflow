Отличный кейс — и он очень важен для управляемости: баг есть, но **контракта нет**, значит сначала нужно **узаконить ожидание**, иначе “баг” превращается в спор.

В режиме **один PR** это делается чисто и контролируемо: PR начинается как **plan + spec-draft**, команда утверждает (Plan Gate), и только потом агент делает фикс — всё в одном PR.

---

# Кейc — Bug без соответствующего R-XXX в spec.md

## Сценарий

Есть наблюдаемая проблема (“что-то не так”), но в `spec.md` нет требования, которое однозначно говорит, что это баг.

Цель: в итоге получить:

* новое/уточнённое требование `R-0YY` в `spec.md`
* план `tasks/T-XXXX.md`
* фикс в коде + тесты
* PR, объясняющий “зачем” и “что меняем”

---

## Шаг 1 — Issue (Backlog)

**Title (первичный):**

* `fix(area): <симптом>`
  пример: `fix(memory): agent sometimes reuses previous task context`

**Labels:** `status:backlog`

**Issue body (минимум):**

```md
Spec: Unknown
Why:
- Это мешает предсказуемости / вызывает инциденты / ломает тесты

Observed:
- ...

Expected:
- ...

Repro:
- ...
```

> Важно: мы не называем это “нарушением контракта” — пока контракта нет.

---

## Шаг 2 — Агент готовит PR (Plan + Spec commit) — без кода

### Ветка

* `fix/XXXX-context-leak`

### Коммит #1 (plan + spec proposal)

В PR первым коммитом агент добавляет/изменяет только документы:

1. `tasks/T-XXXX.md` — план расследования и фикса
2. `spec.md` — добавляет новый `R-0YY` (или уточняет существующий близкий пункт)

**Commit message:**

* `plan(memory): T-XXXX investigate and fix context leak`
  и отдельным коммитом (лучше так):
* `spec(memory): add R-0YY task boundary isolation`

### Что писать в новом requirement (пример)

В `spec.md` в конце:

```md
## R-0YY
Agent MUST NOT reuse prior task state across tasks unless explicitly persisted via LTM tool.
```

Это “узаконивает” ожидание, превращая проблему в проверяемый баг.

---

## Шаг 3 — PR открывается сразу (Plan Gate)

**PR body:**

```md
Fixes #XXXX
Spec: R-0YY
Plan: tasks/T-XXXX.md

Why:
- Сейчас поведение непредсказуемо и ломает доверие к агенту
- Нужна формализация “границы задачи”, чтобы тестировать и не регрессировать
```

**Issue всё ещё:** `status:backlog` (план не утверждён).

---

## Шаг 4 — Plan Gate = утверждение spec + плана

**Команда смотрит только:**

* `tasks/T-XXXX.md` (план: что будем делать)
* `spec.md` (новый `R-0YY`: согласны ли, что это “правильное” ожидаемое поведение)

**Действие команды:** поставить label `plan:approved` на PR
(или оставить `Plan approved ✅`)

После этого:

* issue получает `status:todo` (утверждённый план)
* можно сразу перевести в `status:in-progress` (если агент уже начинает реализацию)

---

## Шаг 5 — Implement phase (теперь можно код)

Теперь агент делает расследование и фикс, следуя плану.

### Рекомендуемая последовательность коммитов (для контроля)

1. Коммит, который добавляет тест, воспроизводящий баг:

   * `test(memory): R-0YY add regression for task boundary isolation`
2. Коммит фикса:

   * `fix(memory): R-0YY clear task state at boundary`
3. Коммит “hardening” (если нужно):

   * `fix(memory): R-0YY prevent implicit state carryover`

> Это максимально прозрачно: сперва “доказали”, потом “починили”.

Если в ходе выясняется, что R-0YY надо уточнить — делаем отдельный коммит:

* `spec(memory): refine R-0YY wording for edge case ...`
  и **можно потребовать re-approve**, если это существенно (снять `plan:approved`).

---

## Шаг 6 — Review → Merge

Команда проверяет:

* тесты зелёные
* фикс соответствует `R-0YY`
* изменения `spec.md` адекватны и не слишком широкие

Merge → issue закрывается → `status:done`.

---

# Важные правила (чтобы кейс не ломал систему)

### Правило 1: “Bug без spec” = “spec-first внутри того же PR”

Никакого кода до Plan Gate — иначе агент начинает “исправлять” непонятно что.

### Правило 2: spec формулирует контракт, не реализацию

`R-0YY` должен описывать требование, а не “сделать reset() в таком-то методе”.

### Правило 3: если не получается сформулировать контракт — это не bug, а idea/spec work

Тогда PR может остаться только plan/spec, а issue закрывается как “resolved by spec clarification”.
